# Безопасность

Проект не изображает из себя отдельную платформу безопасности, но несколько вещей здесь уже сделаны жёстко и без двусмысленности. Это важно именно потому, что внешний интерфейс остаётся простым и дружелюбным.

## Внутренний транспорт

Связи между службами защищены служебными метаданными:

- `bot` и `backend` общаются по внутреннему gRPC с токеном;
- `backend` и `ai-service` используют отдельный токен;
- `correlation_id` проходит через внутренние вызовы и логи;
- при необходимости передаётся контекст действующего лица для аудита.

Неавторизованный внутренний вызов не должен доходить до продуктовой логики.

## Вложения

Для вложений действуют явные ограничения:

- принимаются только поддерживаемые типы;
- у каждого типа есть предел по размеру;
- исполняемые документы и опасные MIME-типы отсекаются;
- имена файлов и метаданные нормализуются;
- путь хранения не может выйти за пределы корневого каталога assets.

## Выгрузки

Для выгрузок важны три правила:

- CSV экранирует значения, похожие на формулы;
- HTML не пытается встраивать небезопасные изображения;
- внутренние заметки попадают в отчёт только при явной настройке.

Это не делает экспорт «абсолютно безопасным» в любой среде, но убирает типичные и наиболее неприятные ошибки.

## Инвайт-коды операторов

Инвайт-коды устроены так:

- код одноразовый;
- в базе хранится не исходное значение, а хэш;
- срок действия ограничен;
- после использования код выключается;
- выпуск доступен только `super_admin`.

## Что лучше не размазывать по интерфейсу

Есть правила, которые должны оставаться жёсткими вне зависимости от того, насколько мягко выглядит Telegram-экран:

- авторизация ролей;
- внутренняя авторизация служб;
- политика вложений;
- жизненный цикл инвайт-кодов;
- аудит чувствительных действий.

Если изменение затрагивает одну из этих зон, правильнее добавить узкое правило в `application` или `infrastructure`, чем пытаться решить вопрос только на уровне `bot`.
