# Безопасность И Hardening

## Что Уже Усилено

Reliora не пытается выглядеть security-platform, но текущий контур уже закрывает базовые эксплуатационные риски, которые особенно важны для Telegram-first helpdesk.

### Internal Transport

- bot и backend общаются через internal gRPC auth metadata;
- backend не пускает неавторизованные вызовы в бизнес-логику;
- correlation id проходит через transport и логи;
- actor context сохраняется там, где нужен audit trail.

### Вложения

На приёме вложений действуют явные ограничения:

- поддерживаются только фото, документы, голосовые и видео;
- по каждому типу действует отдельный лимит размера;
- документы с исполняемыми расширениями или опасными MIME type отклоняются;
- filename и MIME metadata нормализуются перед сохранением;
- storage path не может выйти за пределы asset root.

### Экспорты

- CSV экранирует formula-like значения;
- HTML export не пытается встраивать небезопасные изображения;
- локальные asset path проходят через safe resolver;
- internal notes включаются в ticket report только при явном конфиге.

### Invite-Коды Операторов

- код генерируется как одноразовый;
- в базе хранится hash, а не исходное значение;
- срок действия ограничен;
- после использования код деактивируется;
- доступ к генерации остаётся только у super-admin.

## Что Считать Границей Системы

Telegram UX может оставаться мягким и спокойным, но security-sensitive правила должны оставаться жёсткими и явными:

- авторизация ролей;
- transport auth;
- attachment policy;
- invite lifecycle;
- audit logging.

Если capability затрагивает эти зоны, лучше добавить узкое правило в `application` или `infrastructure`, чем пытаться решить задачу только presentation-слоем.
