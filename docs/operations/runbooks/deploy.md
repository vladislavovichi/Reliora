# Деплой и обновление

## Перед началом

Перед обновлением имеет смысл проверить три вещи:

- актуален ли `.env`;
- не менялись ли внутренние токены без необходимости;
- понятен ли порядок миграций для текущего релиза.

## Обычный порядок

Безопасная последовательность выглядит так:

1. поднять или обновить стек;
2. дождаться готовности служб;
3. прогнать дымовую проверку;
4. только после этого считать релиз принятым.

Практически это обычно выглядит так:

```bash
make up
make health
make smoke
```

## Что происходит при старте

- `ai-service` запускается без миграций;
- `backend` выполняет `alembic upgrade head`;
- `bot` миграции не запускает;
- если `backend` не может дойти до PostgreSQL, Redis или `ai-service`, он завершается с ошибкой;
- если `bot` не может дойти до `backend`, он не остаётся в ложной готовности.

## Производственный запуск без dev-файла

Обычный `make up` добавляет поверх базового Compose-файла `ops/docker/compose.dev.yml`. Для запуска без локальных монтирований используйте только основной файл:

```bash
docker compose -f ops/docker/compose.yml up --build -d
```

## После обновления

Минимальная проверка после релиза:

```bash
make health
make smoke
```

Если обе команды прошли, значит:

- контейнеры поднялись;
- внутренние зависимости доступны;
- `backend` отвечает по gRPC;
- `bot` прошёл свою внутреннюю диагностику.

## Откат

Автоматического rollback в проекте нет. Перед откатом нужно отдельно понять:

- были ли применены миграции;
- совместим ли старый код с уже обновлённой схемой;
- не завязан ли новый релиз на новые данные или индексы;
- не менялись ли внутренние токены и порты.

Если обновление не прошло `smoke`, первым делом смотрите `make logs-backend`: именно `backend` запускает миграции и чаще всего даёт наиболее полезную причину сбоя.
